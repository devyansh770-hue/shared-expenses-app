## 2. Concrete Cases of AI Mistakes & Corrections

### Case 1: Prisma v7 schema.prisma Connection URL
* **AI Mistake**: The AI generated a standard `schema.prisma` containing `url = "file:./dev.db"` inside the `datasource db` block.
* **How Identified**: Running `npx prisma generate` resulted in a validation error:
  ```bash
  Error: Prisma schema validation - (get-config wasm)
  Error code: P1012
  error: The datasource property `url` is no longer supported in schema files. Move connection URLs for Migrate to `prisma.config.ts`...
  ```
* **Correction**: We modified `schema.prisma` to remove `url` from the datasource block. We moved the SQLite database file path declaration to the generated `.env` and `prisma.config.ts` configuration files, which is strict protocol in Prisma v7.8.0.

### Case 2: Implicit 'any' Type Errors in TypeScript App Page
* **AI Mistake**: The AI wrote vanilla-style callback functions in arrays, such as `map(m => m.trim())` and `forEach(d => { ... })` inside the React component `app/page.tsx`.
* **How Identified**: Executing `npm run build` failed during strict TypeScript check stage:
  ```bash
  ./app/page.tsx:368:57
  Type error: Parameter 'm' implicitly has an 'any' type.
  ```
* **Correction**: We edited `app/page.tsx` and annotated all callback parameters with explicit types (e.g. `map((m: string) => m.trim())` and `forEach((d: string) => { ... })`) to satisfy the strict TypeScript compiler configuration generated by Next.js.

---

### Case 3: Node `fs` and `path` Bundle Isolation in Server Actions
* **AI Mistake**: The AI initially planned to import standard Node.js filesystem modules `fs` and `path` at the top level of Server Actions.
* **How Identified**: Standard bundlers (Webpack/Turbopack) in Next.js attempt to compile imports for the client bundle. Top-level Node.js imports in shared actions files can trigger bundling errors indicating that the client cannot resolve `fs`.
* **Correction**: We encapsulated the filesystem reads inside a Server Action using dynamic runtime `require` statements:
  ```typescript
  const fs = require('fs')
  const path = require('path')
  ```
  This guarantees that Node.js native filesystem modules are only loaded during server-side execution, preventing client compilation issues.
